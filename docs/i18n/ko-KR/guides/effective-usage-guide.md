# 효과적인 사용 패턴

이 가이드는 모든 Claude Code 사용자가 첫날부터 알아야 할 필수 패턴을 다룹니다. 공식 [How Claude Code works](https://code.claude.com/docs/en/how-claude-code-works) 및 [Best practices](https://code.claude.com/docs/en/best-practices) 문서를 기반으로 작성되었습니다.

## 가장 중요한 제약: 컨텍스트 윈도우

Claude의 컨텍스트 윈도우에는 대화 내용, 파일 내용, 명령어 출력, CLAUDE.md, 시스템 지침이 담깁니다. 빠르게 채워지며, 채워질수록 성능이 저하됩니다 -- Claude가 이전 지침을 "잊어버리거나" 더 많은 실수를 할 수 있습니다.

설정이 중요한 이유가 바로 여기에 있습니다:

- 잘 작성된 CLAUDE.md는 낭비되는 컨텍스트를 줄입니다 (수정 횟수 감소)
- 좋은 세션 습관은 컨텍스트를 깨끗하게 유지합니다 (아래 세션 관리 참조)
- `/clear`를 언제 사용해야 하는지 알면 성능 저하를 방지할 수 있습니다

## 가장 중요한 습관: Claude가 스스로 작업을 검증할 수 있게 하기

CLAUDE.md에 테스트 명령어, 린트 명령어, 빌드 명령어를 포함시켜 Claude가 스스로 검증할 수 있도록 하세요:

```markdown
## Testing
npm test             # run full test suite
npm run lint         # check for style issues
npm run build        # verify TypeScript compiles
```

프롬프트를 작성할 때 검증 기준을 제공하세요: 예상 출력, 테스트 케이스, 스크린샷 등. Claude는 그럴듯해 보이는 출력에 의존하는 것보다 자체적으로 작업을 검증할 수 있을 때 훨씬 더 나은 결과를 만들어냅니다.

공식 모범 사례에 따르면, 이것이 가장 효과적인 단일 조치입니다.

## 권장 워크플로우

비교적 복잡한 작업에는 다음 사이클을 따르세요:

1. **탐색** -- Claude에게 관련 파일을 읽고 현재 상태를 파악하도록 요청합니다
2. **계획** -- Plan Mode를 사용하여 코딩 전에 계획을 세웁니다
3. **구현** -- Normal Mode로 전환하여 계획을 실행합니다
4. **커밋** -- 변경 사항을 검토하고 커밋합니다

**Plan Mode:** `Shift+Tab`을 두 번 눌러 Plan Mode에 진입합니다. Claude는 읽기 전용 도구를 사용하여 탐색하고 승인을 위한 구현 계획을 만듭니다. 계획을 검토한 후 Normal Mode로 돌아가 실행합니다.

신뢰할 수 있는 에이전트 프레임워크에서 Plan Mode가 갖는 전략적 의미는 [신뢰할 수 있는 에이전트 가이드 § Plan Mode as Strategy-Level Oversight](trustworthy-agents-guide.md#plan-mode-as-strategy-level-oversight)를 참조하세요.

**간단한 작업은 계획을 건너뛰세요** -- 오타 수정, 로그 라인 추가, 단순 이름 변경 등. 작은 변경에는 계획이 오히려 부담이 됩니다.

## 세션 관리 핵심

| 명령어 | 기능 |
| --------- | ------------- |
| `Esc` | 작업 중인 Claude를 중단합니다. 컨텍스트는 유지됩니다. |
| `Esc` 두 번 | 되감기 메뉴를 엽니다 -- 대화, 코드, 또는 둘 다를 체크포인트로 복원합니다 |
| `/rewind` | 더블 `Esc`와 동일 -- 되감기 메뉴를 엽니다 |
| `/clear` | 관련 없는 작업 사이에 컨텍스트를 초기화합니다. **자주 사용하세요.** |
| `/compact` | 대화를 요약하여 컨텍스트를 확보합니다. 초점 추가 가능: `/compact focus on the API changes` |
| `#` | 프롬프트 앞에 `#`를 붙이면 Claude가 해당 학습 내용을 CLAUDE.md에 추가할 것을 제안합니다 (작성 전 사용자가 확인). |
| `/context` | 컨텍스트 윈도우에서 무엇이 공간을 차지하는지 확인합니다. 컨텍스트가 가득 찰 때 진단에 사용합니다. |
| `--continue` | 가장 최근 대화를 이어서 진행합니다 (실행 플래그) |
| `--resume` | 최근 대화 목록에서 선택하여 이어갑니다 (실행 플래그) |
| `/btw` | 곁다리 질문 — 답변이 닫기 가능한 오버레이로 표시되며 대화 기록에 남지 않습니다. 컨텍스트를 늘리지 않고 세부 사항을 확인할 때 사용합니다. |
| `/rename` | 현재 세션에 이름을 붙입니다 (세션을 브랜치처럼 관리). `claude --resume` 실행 시 의미 있는 레이블이 표시됩니다. |
| `Ctrl+G` (plan mode에서) | plan mode 실행 전에 현재 계획을 텍스트 편집기에서 직접 수정합니다. |
| `Esc + Esc` → **Summarize from here** | 부분 압축 — 체크포인트를 선택하여 이후 메시지만 압축하고, 이전 컨텍스트는 그대로 유지합니다. |

**가장 적게 사용되는 명령어는 `/clear`입니다.** 하나의 작업을 마치고 다른 작업을 시작할 때 컨텍스트를 초기화하세요. 이전 작업에서 남은 컨텍스트는 Claude를 혼란스럽게 하고 공간을 낭비합니다.

## 권한 모드

`Shift+Tab`으로 세 가지 모드를 순환합니다:

| 모드 | 동작 |
| ------ | ---------- |
| **Default** | Claude가 편집과 명령 실행 전에 확인을 요청합니다 |
| **Auto-accept edits** | Claude가 자유롭게 파일을 편집하고, 명령 실행 시에만 확인을 요청합니다 |
| **Plan mode** | 읽기 전용 도구만 사용합니다. 실행 전에 승인할 계획을 작성합니다. |

Default 모드로 시작하세요. 작업이 저위험이라고 판단되면 Auto-accept로 전환하세요. 접근 방식을 먼저 검토하고 싶은 복잡한 작업에는 Plan mode를 사용하세요.

## 효과적인 프롬프트 작성법

**처음부터 구체적으로 작성하세요.** 파일을 참조하고, 제약 조건을 언급하고, 패턴을 가리키세요:

```text
Refactor src/api/tasks.ts to use the asyncHandler wrapper
from src/api/middleware.ts. Follow the pattern in src/api/users.ts.
```

**위임하되 지시하지 마세요.** 맥락과 방향을 제공하고, 구현 세부 사항은 Claude에게 맡기세요. 모든 단계를 과도하게 지정하면 시간과 Claude의 컨텍스트가 낭비됩니다.

**풍부한 콘텐츠를 제공하세요.** `@`를 사용하여 파일을 참조하고, 오류나 디자인의 이미지를 붙여넣고, `cat error.log | claude`로 데이터를 파이프하세요. Claude가 처음부터 관련 컨텍스트를 많이 가질수록 수정을 위한 반복이 줄어듭니다.

## 기존 프로젝트에 Claude Code 도입하기

1. **기존 도구를 먼저 확인하세요** -- 린터 설정, 테스트 프레임워크, 빌드 도구를 확인하세요. 해당 명령어를 CLAUDE.md에 추가하세요.
2. **`/init` 또는 `/guardians-of-the-claude:create`를 사용하세요** -- 둘 다 기존 프로젝트 구조를 감지합니다. 프롬프트에서 "Existing project"를 선택하세요.
3. **점진적으로 확장하세요** -- `CLAUDE.md` + `settings.json`으로 시작하세요. 반복적인 필요가 발생할 때만 rules, hooks, agents, skills를 추가하세요.

## 흔한 실패 패턴

| 패턴 | 왜 문제인가 | 해결책 |
| --------- | -------------- | ----- |
| **잡다한 세션** — 관련 없는 작업이 하나의 컨텍스트를 공유 | 작업 A의 컨텍스트가 작업 B를 혼란스럽게 함 | 작업 사이에 `/clear` 사용 |
| **반복적인 수정** | 실패한 시도가 컨텍스트를 노이즈로 오염시킴 | 두 번 실패하면 `/clear` 후, 무엇이 잘못됐는지 반영한 더 나은 프롬프트 작성 |
| **과도하게 지정된 CLAUDE.md** | 긴 파일이 Claude의 주의를 분산시킴 | 과감하게 줄이거나 [규칙 파일](rules-guide.md)로 분리 |
| **신뢰 후 검증 격차** | 그럴듯해 보이는 출력이 정확한 출력과 같지 않음 | 검증 기준을 제공하고, CLAUDE.md에 테스트 명령어를 포함하여 Claude가 자체 검증하도록 함 |
| **끝없는 탐색** | 범위 없는 "코드베이스 조사" 요청이 수십 개 파일을 읽고 컨텍스트를 소진시킴 | 범위를 좁히세요: "토큰 만료 처리를 위해 `src/auth/`만 확인하라" |
