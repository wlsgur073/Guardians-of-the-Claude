# .claude/rules/ の使い方

ルールファイルを使うと、プロジェクトの指示を「単一トピック」のモジュールに分割できます。1 つの大きな CLAUDE.md の代わりに、読みやすくレビューしやすく、コードベースの特定部分にだけスコープを絞れる小さなファイル群を維持する形になります。

## CLAUDE.md とルールの使い分け

**CLAUDE.md を使う対象:** 毎セッションで必要となる中核的な指示 -- ビルドコマンド、プロジェクト概要、テスト手順、主要なアーキテクチャ判断など。

**`.claude/rules/` を使う対象:** モジュール化されたトピック単位の指示。特に次のような場合に有効です。

- CLAUDE.md が 200 行を超えてきて分割したいとき
- 特定のファイルタイプやディレクトリにのみ適用したい指示があるとき
- チームメンバーが領域ごとに分担している場合（フロントエンドルール、バックエンドルール、テストルールなど）
- モノリシックなファイルを編集することなく、トピックを追加・削除したいとき

良い分割の例: CLAUDE.md には本質（200 行未満）を、ルールファイルには詳細を置きます。

## ファイル構成

1 ファイル 1 トピックを基本とします。内容がひと目で分かるように、説明的なファイル名を付けてください。

```text
.claude/rules/
  code-style.md          # 命名、フォーマット、import の規約
  testing.md             # テストフレームワーク、パターン、カバレッジ目標
  architecture.md        # レイヤー間の依存とモジュール境界
  workflow.md            # 開発前チェックリストとレビューゲート
  api-design.md          # API エンドポイントの規約
  database.md            # クエリパターン、マイグレーションのルール
  security.md            # 認証、入力検証、シークレットの取り扱い
```

大規模なプロジェクトでは、ルールをサブディレクトリに整理してください。

```text
.claude/rules/
  frontend/
    components.md        # React コンポーネントのパターン
    styling.md           # CSS / Tailwind の規約
  backend/
    api-handlers.md      # Express ルートハンドラのルール
    database.md          # PostgreSQL のクエリパターン
  shared/
    error-handling.md    # 横断的なエラーハンドリングの規約
```

`paths` フロントマターを持たないルールファイルは、CLAUDE.md と同様に毎セッションでロードされます。簡潔に保ってください -- 目安として、常時ロードされる合計を CLAUDE.md に推奨される ~200 行のバジェット以内に保ってください。

## パススコープ

`paths` フロントマターブロックを追加すると、Claude が指定パターンに一致するファイルを読んだときにのみ、そのルールファイルがロードされるようになります。

```markdown
---
paths:
  - "src/api/**/*.ts"
---
# API エンドポイントのルール
- すべてのエンドポイントは Zod スキーマで入力を検証する
- すべてのルートハンドラで asyncHandler ラッパーを使用する
```

パススコープ付きルールは毎セッションではなく、必要に応じてロードされます。これによりコンテキストがクリーンに保たれます -- Claude は API ファイルを扱うときにのみ API のルールを目にすることになります。

### Glob パターンのリファレンス

| パターン | マッチするもの |
| -------- | -------------- |
| `**/*.ts` | 任意のディレクトリにあるすべての TypeScript ファイル |
| `src/**/*` | `src/` 以下のすべてのファイル |
| `*.md` | プロジェクトルートにある Markdown ファイルのみ |
| `src/components/*.tsx` | 特定のディレクトリ内の React コンポーネント |
| `src/**/*.{ts,tsx}` | 複数の拡張子を扱うブレース展開 |

`paths` には複数のパターンを並べられます。いずれかのパターンにマッチすればそのルールがロードされます。

```yaml
---
paths:
  - "src/api/**/*.ts"
  - "src/middleware/**/*.ts"
---
```

完全なパススコープ付きルールの実例は `templates/advanced/.claude/rules/api-endpoints.md` を参照してください。

## ユーザーレベルのルール

個人用のルールファイルは `~/.claude/rules/` に置くことで、すべてのプロジェクトに適用できます。

```text
~/.claude/rules/
  personal-style.md      # 自分が好むコーディング規約
  git-workflow.md        # 自分のコミットメッセージとブランチ運用の好み
```

ユーザーレベルのルールはプロジェクトのルールよりも先にロードされます。両者が衝突する場合、**プロジェクトルールが優先されます** -- チームの規約が個人の好みを上書きします。

これは、プロジェクト固有ではなく純粋に個人的な好み（エディタ設定のようなもの、好みのコメントスタイルなど）に有用です。

## プロジェクト間でルールを共有する

シンボリックリンクを使うと、ルールファイルを複製せずにプロジェクト間で共有できます。

```bash
# ルールのディレクトリをまるごと共有
ln -s ~/shared-claude-rules .claude/rules/shared

# 単一ファイルを共有
ln -s ~/company-standards/security.md .claude/rules/security.md
```

このパターンは組織全体の標準を扱うのに適しています。中央のリポジトリでルールファイルを管理し、各プロジェクトにシンボリックリンクで取り込みます。中央のルールが更新されると、すべてのプロジェクトが自動的に変更を取り込みます。

**注意:** シンボリックリンクのルールは読み取り時に解決されます。リンク先がすべての開発者のマシンに存在することを確認するか、セットアップスクリプトで作成するようにしてください。
