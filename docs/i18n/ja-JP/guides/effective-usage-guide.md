# 効果的な使い方のパターン

このガイドでは、すべての Claude Code ユーザーが初日から知っておくべき本質的なパターンを扱います。公式ドキュメントの [How Claude Code works](https://code.claude.com/docs/en/how-claude-code-works) と [Best practices](https://code.claude.com/docs/en/best-practices) を典拠としています。

## 最大の制約: コンテキストウィンドウ

Claude のコンテキストウィンドウには、会話・ファイル内容・コマンド出力・CLAUDE.md・システム指示が保持されます。これはあっという間に埋まり、満たされていくにつれてパフォーマンスは劣化します -- Claude は以前の指示を「忘れたり」、ミスが増えたりします。

だからこそ設定が重要なのです。

- よく書かれた CLAUDE.md は無駄なコンテキストを減らします（修正の往復が減ります）
- 良いセッション習慣はコンテキストをクリーンに保ちます（後述の「セッション管理」を参照）
- いつ `/clear` を使うかを知っていれば劣化を防げます

## 最重要のプラクティス: Claude に検証手段を与える

CLAUDE.md にテストコマンド・lint コマンド・ビルドコマンドを記載して、Claude が自己チェックできるようにしましょう。

```markdown
## Testing
npm test             # 全テストスイートを実行
npm run lint         # スタイル違反をチェック
npm run build        # TypeScript のコンパイルを検証
```

プロンプトを書くときは検証基準も渡してください: 期待される出力、テストケース、スクリーンショットなど。Claude は「もっともらしく見える出力」に頼るのではなく、自分で検証できる場合に圧倒的に良い結果を出します。

公式のベストプラクティスによれば、これは単一で最もレバレッジの高い対策です。

## 推奨ワークフロー

非自明なタスクには次のサイクルを使ってください。

1. **探索** -- 関連ファイルを読み、現在の状態を理解するよう Claude に依頼する
2. **計画** -- Plan Mode で実装計画を作成する
3. **実装** -- Normal Mode に切り替えて計画を実行する
4. **コミット** -- 変更をレビューしてコミットする

**Plan Mode:** `Shift+Tab` を 2 回押すと Plan Mode に入ります。Claude は読み取り専用のツールだけを使って探索し、承認用の実装計画を作成します。計画をレビューしてから Normal Mode に戻して実行してください。

信頼できるエージェントフレームワークにおける Plan Mode の戦略的意味については、[信頼できるエージェントガイド § Plan Mode as Strategy-Level Oversight](trustworthy-agents-guide.md#plan-mode-as-strategy-level-oversight) を参照してください。

**自明なタスクでは計画を省略してください** -- タイポ修正、ログ行の追加、単純なリネームなど。小さな変更には計画のオーバーヘッドは見合いません。

## セッション管理の必須項目

| コマンド | 内容 |
| ------- | ---- |
| `Esc` | 動作中の Claude を中断します。コンテキストは保持されます。 |
| `Esc` を 2 回 | 巻き戻しメニューを開きます -- 会話、コード、または両方を任意のチェックポイントに戻せます |
| `/rewind` | `Esc` を 2 回押すのと同じ -- 巻き戻しメニューを開く |
| `/clear` | 関連のないタスクの間でコンテキストをリセットします。**頻繁に使ってください。** |
| `/compact` | 会話を要約してコンテキストを解放します。フォーカスを指定できます: `/compact focus on the API changes` |
| `#` | 現在のセッションで得た学びを CLAUDE.md に追記するよう Claude に依頼します |
| `/context` | 何がコンテキストウィンドウを占有しているかを表示します。コンテキストが満杯に近づいたときの診断に使えます。 |
| `--continue` | 直近の会話を再開します（起動フラグ） |
| `--resume` | 最近の会話の中から再開するものを選びます（起動フラグ） |

**最も使われていないコマンドは `/clear` です。** 1 つのタスクを終えて別のタスクを始めるときは、コンテキストをクリアしてください。前のタスクの残り物は Claude を混乱させ、スペースを無駄にします。

## 権限モード

`Shift+Tab` で 3 つのモードを切り替えられます。

| モード | 動作 |
| ----- | ---- |
| **Default** | Claude は編集とコマンドの前に確認を求めます |
| **Auto-accept edits** | Claude はファイルを自由に編集します。コマンドにはなお確認を求めます |
| **Plan mode** | 読み取り専用のツールのみ。承認用の計画を作成してから実行します。 |

まずは Default モードから始めてください。タスクが低リスクと信じられるようになったら Auto-accept に進んでください。複雑なタスクで、まずアプローチをレビューしたい場合は Plan モードを使ってください。

## 効果的なプロンプトの書き方

**最初から具体的に。** ファイルを参照し、制約を明示し、参考にするパターンを指し示してください。

```text
src/api/middleware.ts の asyncHandler ラッパーを使って
src/api/tasks.ts をリファクタリングしてください。
src/api/users.ts のパターンに従ってください。
```

**指示するのではなく委任する。** コンテキストと方向性を渡し、実装の細部は Claude に決めさせましょう。すべてのステップを過剰に指定すると、あなたの時間も Claude のコンテキストも無駄になります。

**豊かな素材を渡す。** `@` でファイルを参照し、エラーやデザインの画像を貼り付け、`cat error.log | claude` のようにデータをパイプで渡してください。最初に手元にある関連コンテキストが多いほど、修正の往復は少なくなります。

## 既存プロジェクトに Claude Code を導入する

1. **既存のツールをまず探索する** -- linter の設定、テストフレームワーク、ビルドツールを確認してください。それらのコマンドを CLAUDE.md に追加します。
2. **`/init` または `/guardians-of-the-claude:create` を使う** -- どちらも既存のプロジェクト構造を検出します。プロンプトで「Existing project」を選んでください。
3. **少しずつ育てる** -- まずは `CLAUDE.md` と `settings.json` から。ルール・フック・エージェント・スキルは、繰り返し必要になったときにだけ追加してください。

## よくある失敗パターン

### 1. 何でも放り込んだセッション

無関係なタスクを 1 つのセッションに混ぜてしまうケースです。タスク A のコンテキストがタスク B を混乱させます。

**対処:** 無関係なタスクの間で `/clear` を使ってください。1 セッション 1 タスクが原則です。

### 2. 何度も修正を重ねる

修正の繰り返しは、失敗した試行でコンテキストを汚染します。修正のたびにノイズが増えます。

**対処:** 2 回失敗したら `/clear` し、何が間違っていたかを盛り込んだ、より良い初期プロンプトを書き直してください。

### 3. 過剰に書き込まれた CLAUDE.md

長い CLAUDE.md は Claude の注意を薄めます。**対処:** 容赦なく剪定するか、[ルールファイル](rules-guide.md)に分割してください。

### 4. 信頼すれども検証せず

もっともらしく見える出力をそのまま受け入れてしまうケースです。Claude のコードはコンパイルが通っても、エッジケースを取りこぼしているかもしれません。

**対処:** 必ず検証基準を渡してください。CLAUDE.md にテストコマンドを記載し、変更後にテストを実行するよう Claude に頼んでください。

### 5. 際限のない探索

スコープなしに「コードベースを調査して」と頼んでしまうケースです。Claude は何十ものファイルを読み、コンテキストを満たし、フォーカスを失います。

**対処:** スコープを狭めましょう。「コードベースを調査して」ではなく、「`src/auth/` のトークン期限切れ処理だけをチェックして」のように。

## プラグインの学習状態を理解する

`guardians-of-the-claude` プラグインは、フロントエンドのビルドプロセスが最適化された成果物を生成するのと似た方法で状態を管理します：

| ビルドプロセスの段階 | プラグインの対応物 |
|---|---|
| Transpiling（最新 → 互換可能構文） | スキーママイグレーション (v1.0.0 → v1.2.0) |
| Bundling（多数のファイル → 単一バンドル） | `state-summary.md` レンダラ |
| Tree shaking（未使用コードの削除） | コンパクション（>30日 → 四半期ロールアップ） |
| Production optimization (minify) | トークン予算の適用 |

重要な違いは時点です: ビルドは *デプロイ時に一度* だけ起こりますが、プラグインの学習は *呼び出しごとに段階的に* 進みます。より正確なアーキテクチャモデルは **event sourcing** であり、内部の実装は `plugin/references/learning-system.md` を参照してください。
