# 効果的な CLAUDE.md の書き方

## CLAUDE.md とは

CLAUDE.md は、Claude が各セッションの開始時に読み込む永続的な指示を記述した Markdown ファイルです。強制される設定ではなく、Claude のふるまいを形づくる「コンテキスト」です。ブリーフィング資料のようなものだと考えてください -- 書き方が良いほど、Claude はあなたのプロジェクトでより良いパフォーマンスを発揮します。

## 階層構造

Claude は複数の場所から指示を読み込みます。スコープがより限定的な場所が、より広いスコープよりも優先されます。

| スコープ | 場所 | 用途 |
| ------- | --- | ---- |
| 管理ポリシー | プラットフォーム固有のシステムパス | 管理者が設定する組織全体の指示 |
| プロジェクト | `./CLAUDE.md` または `./.claude/CLAUDE.md` | チームで共有するプロジェクト指示（git にコミット） |
| ユーザー | `~/.claude/CLAUDE.md` | すべてのプロジェクトに適用される個人の好み |

指示が衝突する場合は、より限定的な場所が優先されます。プロジェクトレベルのルールはユーザーレベルの好みを上書きします。組織が設定した管理ポリシーが最も高い優先度を持ちます。

## プロジェクト指示を置ける 2 つの場所

プロジェクトの CLAUDE.md は、以下の 2 つの場所のいずれかに配置できます。

- **`./CLAUDE.md`**（プロジェクトルート） -- ひと目で見えます。リポジトリを閲覧する人がすぐに気づきます。
- **`./.claude/CLAUDE.md`** -- プロジェクトルートをすっきり保てます。ルートにすでに多くの設定ファイルがあるリポジトリに向いています。

**両方ではなく、どちらか一方を選んでください。** 両方が存在する場合 Claude はどちらも読み込み、指示が衝突する可能性があります。ルートのほうが一般的で、`/init` が生成するのもこちらです。

## フォルダレベルの CLAUDE.md

任意のサブディレクトリにも CLAUDE.md を配置できます。これらのファイルは遅延ロードされます。Claude がそのディレクトリ内のファイルにアクセスしたときに初めて読み込まれます。

プロジェクトの一部分にのみ固有のコンテキストには、フォルダレベルのファイルを使ってください。

- `src/CLAUDE.md` -- ソースコードの規約
- `tests/CLAUDE.md` -- テストのパターンとヘルパー
- `docs/CLAUDE.md` -- ドキュメンテーションの基準

これにより、ルートの CLAUDE.md はプロジェクト全体の指示に集中させつつ、より深いコンテキストを必要な場所だけに提供できます。あるいは、パススコープ付きの指示には `.claude/rules/` ファイルを使う方法もあります -- [Rules ガイド](rules-guide.md) を参照してください。

## 執筆の原則

- **200 行未満を目指す。** これは厳密な上限ではなく、ゆるい目安です。Claude は長さに関係なくファイル全体を読み込みますが、短いほうが指示への準拠度は高くなります。
- **Markdown のヘッダーと箇条書きを使う。** 構造があると、Claude にとっても人間にとってもスキャンしやすくなります。
- **具体的に、検証可能に書く。** 「コードを適切にフォーマットする」ではなく「2 スペースインデントを使う」と書きましょう。「動くようにする」ではなく「`npm test` で検証する」と書きましょう。
- **矛盾する指示を避ける。** CLAUDE.md とルールファイルが異なることを言っていると、Claude はどちらを採用するか分かりません。すべてのファイルをまとめて見直し、矛盾を検出してください。

## 含めるものと除外するもの

これは CLAUDE.md を書くうえで最も重要な判断です。

| 含める | 除外する |
| ------ | ------- |
| Claude が推測できない bash コマンド | コードを読めば Claude が分かること |
| デフォルトと異なるコードスタイルのルール | Claude がすでに知っている標準的な言語規約 |
| テストの指示と推奨するテストランナー | 詳細な API ドキュメント（代わりにドキュメントへリンクする） |
| リポジトリのエチケット（ブランチ名、PR の規約） | 頻繁に変わる情報 |
| プロジェクト固有のアーキテクチャ判断 | 長い解説やチュートリアル |
| 開発環境のクセ（必要な環境変数、サービス） | コードベースのファイルごとの説明 |
| よくある落とし穴や非自明なふるまい | 「クリーンなコードを書く」のような自明な作法 |

判断の目安: **その情報がなければ Claude がミスをするか?** ミスをするなら含めましょう。Claude が自力で気づくならば、書かないでおきましょう。

## @import 構文

外部ファイルを参照することで、CLAUDE.md を簡潔に保ちつつ、より深いコンテキストへとリンクできます。

```markdown
## References
プロジェクト概要は @README.md を参照
@docs/architecture.md
@docs/api-conventions.md
```

要点:

- **相対パス**は `@import` を含むファイルから解決されます: `@docs/guide.md`
- **絶対パス**はファイルシステムのルートから始まります: `@/home/user/notes.md`
- **個人インポート**はホームディレクトリを参照します: `@~/.claude/my-project-instructions.md`
- **最大の深さ**は 5 ホップです -- インポートされたファイルがさらに別のファイルをインポートでき、最大 5 階層までネストできます。

CLAUDE.md に内容を重複させるのではなく、`@import` を使って既存のドキュメントを Claude に指し示しましょう。

## スケールに応じた整理

プロジェクトに複数の skill や agent がある場合は、各スキルの名前／目的、各エージェントの名前／モデル／役割を一覧にしたクイックリファレンステーブルを CLAUDE.md に追加してください。これにより、Claude はディレクトリを探索しなくても何が利用できるかを把握できます。定義はそれぞれのファイルに残し、CLAUDE.md のテーブルはあくまで参照用として扱いましょう。

## CLAUDE.md を剪定する

CLAUDE.md はコードのように扱ってください -- 定期的にレビューし、積極的に剪定しましょう。

各行について自問してください: **「この行を消したら Claude がミスをするか?」** 答えが「いいえ」ならば、削ってください。肥大化した CLAUDE.md は Claude の注意を多すぎる指示に分散させ、本当に重要なルールがノイズに埋もれてしまいます。

ルールが特に重要な場合は、強調を加えて目立たせましょう。

- 「IMPORTANT: main に直接コミットしないこと」
- 「YOU MUST: コミット前にテストスイートを実行すること」

強調は本当に重要なルールにだけ使ってください。すべてに IMPORTANT を付けると、何も重要ではなくなります。

## よくある間違い

1. **長すぎる** -- 500 行の CLAUDE.md は、各行に対する Claude の注意を薄めます。[ルールファイル](rules-guide.md)に分割するか、剪定してください。
2. **曖昧すぎる** -- 「ベストプラクティスに従う」では Claude に何も伝わりません。「テストデータにはファクトリを使い、インラインのオブジェクトは使わない」と書けば、何をすべきかが正確に伝わります。
3. **指示の矛盾** -- CLAUDE.md は「default export を使う」と言っているのに、ルールファイルは「named export を使う」と言っている。すべてのファイルをまとめて見直してください。
4. **当たり前のことを書く** -- Claude はすでに標準的な言語規約・関数の書き方・一般的なライブラリ API を知っています。あなたのプロジェクト固有のことに集中してください。

## /init というショートカット

ゼロから始める場合は、Claude Code 内で `/init` を実行してください。Claude がコードベースを解析し、出発点となる CLAUDE.md を生成します。これは[ベストプラクティス](https://code.claude.com/docs/en/best-practices)で公式に推奨されているスタート地点です。出力はドラフトとして扱い -- レビューし、テンプレートのセクションをマージし、不要なものを剪定してください。
