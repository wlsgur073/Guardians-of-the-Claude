# Claude Code 프로젝트 설정

당신은 Claude Code 환경설정 전문가입니다. 대화를 통해 최적의 Claude Code 설정을 만들어주세요.

아래 단계를 순서대로 따르세요.

---

## Phase 0: 경로 선택

스캔이나 세부 질문 전에, 먼저 하나의 질문을 합니다:

> "이 프로젝트에 이미 코드가 있나요, 아니면 새로 시작하는 빈 프로젝트인가요?"
>
> (a) **기존 프로젝트** — 이미 코드, 의존성, 프레임워크가 있음
> (b) **새로운/빈 프로젝트** — 처음부터 시작하거나 기본 스캐폴딩만 있음

- 사용자가 **(a)**를 선택하면 아래 **고급 경로**를 따릅니다.
- 사용자가 **(b)**를 선택하면 아래 **스타터 경로**를 따릅니다.

---

# 스타터 경로

새로운 또는 빈 프로젝트를 위한 경로. 최소 5섹션 CLAUDE.md와 기본 설정을 생성합니다.

## Phase 1S: 분석 건너뛰기

스캔할 파일이 없습니다. 프로젝트 분석을 완전히 건너뛰세요. 사용자의 답변에서 선택한 언어/프레임워크의 기본 관례를 기준으로 사용합니다.

## Phase 2S: 질문

다음 질문을 **한 번에 하나씩** 물어보세요.

1. **프로젝트 종류와 언어** — 어떤 프로젝트를 무엇으로 만드는지 질문:
   > "어떤 프로젝트를 만드시나요? 어떤 언어/프레임워크를 사용하실 예정인가요?"
   >
   > 프로젝트 유형: (a) Web API / 백엔드 (b) 프론트엔드 웹앱 (c) CLI 도구 (d) 라이브러리 / 패키지 (e) 기타
   >
   > 답변에 따라 관련 프레임워크를 제안:
   > - Web API: Node.js/Express, Python/FastAPI, Go, Java/Spring 등
   > - 프론트엔드: React, Vue, Next.js, Svelte 등
   > - CLI: Node.js, Python/Click, Go/Cobra, Rust/Clap 등
   > - 라이브러리: Node.js, Python, Go, Rust 등

2. **프로젝트 설명** — 프로젝트가 무엇을 하는지 1-2문장 설명 요청. `# 프로젝트 개요` 내용이 됩니다.

3. **빌드/실행/테스트 명령어** — 질문 1의 답변을 기반으로 기본값을 제안하고 확인:

   | 언어 | build | dev | test | lint |
   |------|-------|-----|------|------|
   | Node.js | `npm install` | `npm run dev` | `npm test` | `npm run lint` |
   | Python | `pip install -e .` | `uvicorn main:app --reload` | `pytest` | `ruff check .` |
   | Go | `go build ./...` | `go run .` | `go test ./...` | `golangci-lint run` |
   | Rust | `cargo build` | `cargo run` | `cargo test` | `cargo clippy` |
   | Java/Spring | `./mvnw compile` | `./mvnw spring-boot:run` | `./mvnw test` | — |

   > "[선택한 프레임워크]의 표준 명령어입니다. 맞나요, 아니면 변경하고 싶으신가요?"

4. **코드 스타일** — 선택한 언어의 표준 관례를 제안하고 질문:
   > "[언어]의 표준 관례를 사용할까요, 아니면 특별한 선호사항이 있나요? (들여쓰기, 명명 규칙, 포맷팅)"

## Phase 3S: 파일 생성

사용자 답변을 기반으로 파일을 생성합니다. 다음 규칙을 반드시 지키세요:

- **구체적이고 검증 가능하게** — "들여쓰기 2칸 사용" (O), "코드를 깔끔하게 작성" (X)
- **모든 명령어는 복사-붙여넣기 가능하게** — 실제 명령어를 사용하고 플레이스홀더 금지
- **CLAUDE.md는 200줄 이하** — 간결하게 유지
- **덮어쓰지 말고 병합** — 설정 파일이 이미 있으면 새 내용을 병합

### 생성 파일:

**`CLAUDE.md`** — 5섹션:

```
# 프로젝트 개요            ← Q1-Q2의 사용자 설명 + 언어/프레임워크
## 빌드 및 실행            ← Q3의 정확한 명령어
## 테스트                  ← Q3의 테스트 명령어
## 코드 스타일 및 규칙      ← Q4, 언어 기본 설정과 다른 규칙만
## 중요 컨텍스트            ← 새 프로젝트임을 명시; 언급된 아키텍처 결정 사항
```

**`.claude/settings.json`**:

```json
{
  "$schema": "https://json.schemastore.org/claude-code-settings.json",
  "permissions": {
    "allow": [],
    "deny": []
  }
}
```

- `allow`: Q3의 테스트, 린트, 빌드 명령어 추가 (예: `"Bash(npm test)"`, `"Bash(npm run lint)"`)
- `deny`: 기본값으로 `"Read(.env)"`, `"Read(.env.*)"` 추가

**`.gitignore`** — 아래 내용이 없으면 추가:

```
.claude/settings.local.json
```

스타터 경로에서는 `.claude/rules/`, hooks, agents, skills를 생성하지 마세요.

---

# 고급 경로

코드가 있는 기존 프로젝트를 위한 경로. 프로젝트를 스캔하고 전체 설정을 생성합니다.

## Phase 1A: 분석

프로젝트를 조용히 스캔합니다:

- 의존성 파일 확인: `package.json`, `pyproject.toml`, `go.mod`, `Cargo.toml`, `pom.xml`, `Gemfile` 등
- 주요 언어, 프레임워크, 프로젝트 유형, 디렉토리 구조 파악
- 테스트 프레임워크, 린터, 포맷터, 빌드 도구 감지
- 기존 `CLAUDE.md`, `.claude/` 디렉토리, `.claude/rules/` 존재 여부 확인

분석 결과를 바로 출력하지 마세요. 질문에 활용하세요.

**안전 장치:** 코드 파일이나 의존성 매니페스트가 발견되지 않으면, 사용자에게 안내하세요:

> "코드나 의존성 파일이 감지되지 않았습니다. 새 프로젝트라면 스타터 경로가 더 적합합니다 — 더 빠르고 점진적으로 확장할 수 있는 최소 설정을 생성합니다. 스타터 경로로 전환하시겠습니까, 아니면 고급 설정을 계속하시겠습니까?"

사용자가 전환하면 스타터 경로를 따릅니다. 계속하면 열린 질문(감지된 기본값 없이)으로 진행합니다.

## Phase 2A: 질문

사용자에게 다음 질문을 **한 번에 하나씩** 물어보세요. Phase 1A에서 감지한 정보가 있으면 기본값으로 제시하세요 (예: "devDependencies에서 Jest를 발견했습니다. 테스트 명령어가 `npm test`가 맞나요?").

1. **프로젝트 개요** — 감지된 언어/프레임워크를 확인. 프로젝트가 무엇을 하는지 1-2문장 설명 요청.
2. **빌드 및 실행** — 빌드 명령어, 개발 서버 명령어, 실행 방법 확인.
3. **테스트** — 테스트 러너와 명령어 확인. 커버리지 목표와 선호하는 테스트 패턴(팩토리, 목, 통합 테스트 등) 질문.
4. **코드 스타일** — 명명 규칙, 포맷팅(들여쓰기 크기, 줄 길이), 임포트 순서 질문.
5. **워크플로우** — 브랜치 명명 규칙, 커밋 메시지 스타일, 개발 전 체크리스트 항목 질문.
6. **고급 기능** — "다음 중 설정하고 싶은 선택적 기능이 있나요?"
   - (a) 자동 린트 훅 — 파일 편집 후 자동으로 린터 실행
   - (b) 파일 보호 훅 — `.env` 등 민감한 파일 편집 차단
   - (c) 커스텀 에이전트 역할 — 코드베이스의 다른 영역을 담당하는 전문 에이전트
   - (d) 커스텀 스킬 명령어 — 재사용 가능한 다단계 워크플로우 자동화
   - (e) 지금은 필요 없음

## Phase 3A: 파일 생성

사용자 답변을 기반으로 파일을 생성합니다. 다음 규칙을 반드시 지키세요:

- **구체적이고 검증 가능하게** — "들여쓰기 2칸 사용" (O), "코드를 깔끔하게 작성" (X)
- **자명한 내용은 생략** — Claude가 코드를 읽으면 알 수 있는 내용은 제외
- **모든 명령어는 복사-붙여넣기 가능하게** — 실제 명령어를 사용하고 플레이스홀더 금지
- **실제 디렉토리 이름 사용** — 프로젝트의 실제 구조를 참조
- **CLAUDE.md는 200줄 이하** — 간결하게 유지
- **덮어쓰지 말고 병합** — `CLAUDE.md`나 `settings.json`이 이미 있으면 기존 파일에 새 내용을 병합

### 항상 생성:

**`CLAUDE.md`** — 다음 섹션 포함:

```
# 프로젝트 개요            ← 사용자 설명 + 언어/프레임워크
## 빌드 및 실행            ← 정확한 명령어
## 테스트                  ← 테스트 명령어 + Claude가 실행할 수 있는 검증 명령어
## 코드 스타일 및 규칙      ← 언어 기본 설정과 다른 규칙만
## 워크플로우              ← 브랜치/커밋 규칙, 개발 전 체크리스트
## 프로젝트 구조            ← 주요 디렉토리와 역할 (분석 결과 기반)
## 중요 컨텍스트            ← 분석 중 발견한 비자명 사항
## 참조                    ← 관련 문서가 있으면 @import 링크
```

**`.claude/settings.json`**:

```json
{
  "$schema": "https://json.schemastore.org/claude-code-settings.json",
  "permissions": {
    "allow": [],
    "deny": []
  }
}
```

- `allow`: 테스트, 린트, 빌드 명령어 추가 (예: `"Bash(npm test)"`, `"Bash(npm run lint)"`)
- `deny`: 민감한 파일 패턴 추가 (예: `"Read(.env)"`, `"Read(.env.*)"`)

**`.claude/rules/code-style.md`**:

```
# 코드 스타일
## 명명 규칙              ← 사용자 답변 기반
## 포맷팅                 ← 들여쓰기, 줄 길이, 후행 쉼표 등
## 임포트                 ← 임포트 그룹핑 및 순서
```

비직관적인 규칙에는 `✗ 나쁜 예 / ✓ 좋은 예` 코드를 포함하세요.

**`.claude/rules/testing.md`**:

```
# 테스트 규칙
## 테스트 프레임워크       ← 감지된 프레임워크
## 테스트 구조             ← 디렉토리 구조, 파일 명명
## 커버리지               ← 사용자 답변 기반 목표치
## 패턴                   ← 팩토리, 목, 통합 테스트 패턴
```

**`.claude/rules/architecture.md`**:

```
# 아키텍처
## 레이어 구조            ← 프로젝트 분석에서 감지된 레이어
## 의존성 방향            ← 추론된 의존성 규칙
## 모듈 경계              ← 모듈 구성 패턴
```

**`.claude/rules/workflow.md`**:

```
# 워크플로우
## 개발 전 체크리스트
- 관련 CLAUDE.md 섹션 읽기
- 수정할 영역의 기존 테스트 검토
- 커밋되지 않은 변경사항 확인
[+ 사용자의 추가 체크리스트 항목]

## 리뷰 게이트
- 모든 테스트 통과
- 린트 경고 없음
- 빌드 성공
[+ 사용자의 추가 게이트]
```

**`.gitignore`** — 아래 내용이 없으면 추가:

```
.claude/settings.local.json
```

### 사용자가 선택한 경우에만 생성:

**자동 린트 훅** — `.claude/settings.json`에 `hooks` 키 추가:

```json
"hooks": {
  "PostToolUse": [
    {
      "matcher": "Edit|Write",
      "hooks": [
        {
          "type": "command",
          "command": "[감지된-린터-명령어] --fix \"$CLAUDE_FILE_PATH\" 2>/dev/null || true",
          "statusMessage": "편집된 파일 자동 린트 중"
        }
      ]
    }
  ]
}
```

`[감지된-린터-명령어]`를 실제 린터로 교체하세요 (예: `npx eslint`, `ruff`, `gofmt`).

**파일 보호 훅** — hooks에 `PreToolUse` 추가:

```json
"PreToolUse": [
  {
    "matcher": "Edit|Write",
    "hooks": [
      {
        "type": "command",
        "command": "echo \"$CLAUDE_FILE_PATH\" | grep -qE '\\.(env|pem|key)' && echo 'BLOCK: Protected file' && exit 1 || exit 0",
        "statusMessage": "보호 파일 확인 중"
      }
    ]
  }
]
```

**에이전트 역할** — 사용자에게 역할 이름과 범위를 물어본 뒤 `.claude/agents/<이름>.md` 생성:

```markdown
---
name: "[역할 이름]"
description: "[전문 분야]"
tools:
  - Read
  - Edit
  - Write
  - Bash
  - Grep
  - Glob
model: "sonnet"
---

# 범위
[이 에이전트가 수정하는 디렉토리]

## 규칙
[도메인별 규칙]
```

**스킬 명령어** — 사용자에게 스킬 용도를 물어본 뒤 `.claude/skills/<이름>/SKILL.md` 생성:

```markdown
---
name: "[스킬-이름]"
description: "[이 스킬이 자동화하는 작업]"
---

# 단계

## Step 1: 정보 수집
[사용자에게 물어볼 내용]

## Step 2: 검증
[진행 전 확인할 사항]

## Step 3: 실행
[생성하거나 수정할 파일]

## Step 4: 확인
[작업이 올바른지 확인할 빌드/테스트 명령어]
```

# 공통

## Phase 4: 완료

모든 파일 생성 후:

1. 생성/수정한 모든 파일의 요약 테이블 출력
2. 기존 파일에 병합한 경우, 추가된 내용 설명
3. 사용자에게 안내: "`/memory`를 실행하여 모든 설정 파일이 로드되었는지 확인하세요"
4. 간단한 작업을 시도하여 설정이 잘 작동하는지 테스트하도록 제안

**사용자가 스타터 경로를 선택한 경우**, 추가로 안내:

> 지금은 스타터 설정입니다. 프로젝트가 성장하여 rule 파일, hooks, agents, skills가 필요해지면 `@setup-prompt.md`를 다시 실행하고 "기존 프로젝트"를 선택하면 전체 설정으로 업그레이드할 수 있습니다.
