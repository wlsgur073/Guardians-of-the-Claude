# 효과적인 CLAUDE.md 파일 작성법

## CLAUDE.md란?

CLAUDE.md는 Claude가 매 세션 시작 시 읽는 지속적인 지시사항을 담고 있는 마크다운 파일입니다. 강제되는 설정이 아니라 Claude의 동작을 형성하는 컨텍스트입니다. 브리핑 문서라고 생각하면 됩니다. 잘 작성할수록 Claude가 프로젝트에서 더 나은 성능을 발휘합니다.

## 계층 구조

Claude는 여러 위치에서 지시사항을 로드하며, 더 구체적인 범위가 더 넓은 범위보다 우선합니다:

| 범위 | 위치 | 용도 |
| ------- | ---------- | --------- |
| 관리 정책 | 플랫폼별 시스템 경로 | 관리자가 설정하는 조직 전체 지시사항 |
| 프로젝트 | `./CLAUDE.md` 또는 `./.claude/CLAUDE.md` | 팀 공유 프로젝트 지시사항 (git에 커밋) |
| 사용자 | `~/.claude/CLAUDE.md` | 모든 프로젝트에 적용되는 개인 환경설정 |

지시사항이 충돌할 경우, 더 구체적인 위치가 우선합니다. 프로젝트 수준의 규칙이 사용자 수준의 환경설정을 덮어씁니다. 조직에서 설정한 관리 정책이 가장 높은 우선순위를 가집니다.

## 프로젝트 지시사항의 두 가지 위치

프로젝트 CLAUDE.md는 다음 두 위치 중 하나에 배치할 수 있습니다:

- **`./CLAUDE.md`** (프로젝트 루트) -- 한눈에 보입니다. 저장소를 탐색하는 누구나 즉시 확인할 수 있습니다.
- **`./.claude/CLAUDE.md`** -- 프로젝트 루트를 더 깔끔하게 유지합니다. 이미 루트 수준에 설정 파일이 많은 저장소에 적합합니다.

**둘 중 하나만 선택하세요.** 둘 다 존재하면 Claude가 모두 로드하며, 지시사항이 충돌할 수 있습니다. 루트 위치가 더 일반적이며 `/init`이 생성하는 위치이기도 합니다.

## 폴더 수준 CLAUDE.md

모든 하위 디렉토리에 CLAUDE.md를 배치할 수 있습니다. 이 파일들은 지연 로드됩니다: Claude는 해당 디렉토리의 파일에 접근할 때만 읽습니다.

프로젝트의 특정 부분에 해당하는 컨텍스트에 폴더 수준 파일을 사용하세요:

- `src/CLAUDE.md` -- 소스 코드 규칙
- `tests/CLAUDE.md` -- 테스트 패턴 및 헬퍼
- `docs/CLAUDE.md` -- 문서 표준

이렇게 하면 루트 CLAUDE.md는 프로젝트 전체 지시사항에 집중하면서, 필요한 곳에 정확히 더 깊은 컨텍스트를 제공할 수 있습니다. 실제 예시는 `examples/src/CLAUDE.md`와 `examples/tests/CLAUDE.md`를 참고하세요.

## 작성 원칙

- **200줄 이하를 목표로 하세요.** 이것은 소프트 가이드라인이지 하드 제한이 아닙니다. Claude는 길이에 관계없이 전체 파일을 로드하지만, 짧은 파일이 지시사항에 대한 더 나은 준수율을 보입니다.
- **마크다운 제목과 글머리 기호를 사용하세요.** 구조화하면 Claude와 사람 모두가 지시사항을 빠르게 파악할 수 있습니다.
- **구체적이고 검증 가능하게 작성하세요.** "코드를 적절히 포맷하라"가 아니라 "2칸 들여쓰기를 사용하라"고 작성하세요. "잘 동작하는지 확인하라"가 아니라 "`npm test`를 실행하여 검증하라"고 작성하세요.
- **충돌하는 지시사항을 피하세요.** CLAUDE.md가 한 가지를 말하고 규칙 파일이 다른 것을 말하면, Claude는 어느 쪽이든 따를 수 있습니다. 모순이 없는지 점검하세요.

## 포함할 것과 제외할 것

CLAUDE.md를 작성할 때 가장 중요한 결정입니다:

| 포함할 것 | 제외할 것 |
| --------- | --------- |
| Claude가 추측할 수 없는 Bash 명령어 | Claude가 코드를 읽으면 알 수 있는 것 |
| 기본값과 다른 코드 스타일 규칙 | Claude가 이미 알고 있는 표준 언어 관례 |
| 테스트 지시사항 및 선호하는 테스트 러너 | 상세한 API 문서 (대신 문서로 링크) |
| 저장소 에티켓 (브랜치 명명, PR 관례) | 자주 변경되는 정보 |
| 프로젝트 고유의 아키텍처 결정 | 긴 설명이나 튜토리얼 |
| 개발 환경 특이사항 (필수 환경 변수, 서비스) | 코드베이스의 파일별 설명 |
| 자주 발생하는 실수나 비직관적 동작 | "깔끔한 코드를 작성하라" 같은 자명한 실천 |

경험 법칙: 이 정보가 없으면 Claude가 실수할 것이라면 포함하세요. Claude가 스스로 알아낼 수 있다면 제외하세요.

## @import 구문

외부 파일을 참조하여 CLAUDE.md를 핵심에 집중하면서 더 깊은 컨텍스트에 연결할 수 있습니다:

```markdown
## References
See @README.md for project overview
@docs/architecture.md
@docs/api-conventions.md
```

주요 세부사항:

- **상대 경로**는 `@import`를 포함하는 파일을 기준으로 해석됩니다: `@docs/guide.md`
- **절대 경로**는 파일 시스템 루트에서 시작합니다: `@/home/user/notes.md`
- **개인 임포트**는 홈 디렉토리를 참조합니다: `@~/.claude/my-project-instructions.md`
- **최대 깊이**는 5단계입니다 -- 임포트된 파일이 다른 파일을 임포트할 수 있으며, 최대 5단계까지 가능합니다.

CLAUDE.md에 내용을 복제하지 말고, `@import`를 사용하여 기존 문서를 가리키세요.

## CLAUDE.md 정리하기

CLAUDE.md를 코드처럼 다루세요 -- 정기적으로 검토하고 적극적으로 정리하세요.

각 줄에 대해 물어보세요: **"이것을 제거하면 Claude가 실수할까?"** 답이 아니오라면, 지우세요. 비대한 CLAUDE.md는 Claude의 주의를 너무 많은 지시사항에 분산시키고, 중요한 규칙이 잡음 속에 묻히게 됩니다.

규칙이 중요할 때는 강조를 추가하여 눈에 띄게 하세요:

- "IMPORTANT: main에 직접 커밋하지 마세요"
- "YOU MUST 커밋 전에 테스트 스위트를 실행하세요"

강조는 정말 중요한 규칙에만 사용하세요. 모든 것이 IMPORTANT로 표시되면, 아무것도 중요하지 않게 됩니다.

## 흔한 실수

1. **너무 긴 경우** -- 500줄짜리 CLAUDE.md는 Claude가 각 줄에 덜 주의를 기울인다는 의미입니다. [규칙 파일](rules-guide.md)로 분리하거나 정리하세요.
2. **너무 모호한 경우** -- "모범 사례를 따르라"는 Claude에게 아무 정보도 주지 않습니다. "테스트 데이터에 팩토리를 사용하고, 인라인 객체는 절대 사용하지 마라"는 정확히 무엇을 해야 하는지 알려줍니다.
3. **충돌하는 지시사항** -- CLAUDE.md는 "default export를 사용하라"고 하고 규칙 파일은 "named export를 사용하라"고 합니다. 모든 파일을 함께 점검하세요.
4. **당연한 것을 언급하는 경우** -- Claude는 이미 표준 언어 관례, 함수 작성법, 일반적인 라이브러리 API를 알고 있습니다. 프로젝트에 고유한 것에 집중하세요.

## /init 단축키

처음부터 시작하는 경우, Claude가 CLAUDE.md를 생성해줄 수 있습니다:

```text
claude
> /init
```

Claude가 코드베이스를 분석하고 발견한 내용을 기반으로 초기 CLAUDE.md를 생성합니다. 이것은 [모범 사례](https://code.claude.com/docs/en/best-practices)에 따른 공식 권장 시작점입니다.

다단계 흐름의 더 대화형 경험을 원한다면, 실행 전에 환경 변수를 설정하세요:

```bash
CLAUDE_CODE_NEW_INIT=true claude
> /init
```

어느 방법이든 출력 결과를 초안으로 취급하세요. 검토하고, 템플릿의 섹션을 병합하고, 불필요한 것은 정리하세요.
